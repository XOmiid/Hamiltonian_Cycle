Hamiltonian Cycle Problem
A deep dive into one of the most famous NP-Complete problems in computer science.
NP-Complete Graph Theory Decision Problem
1. Problem Definition
Formal Definition
Given an undirected graph G = (V, E) where V is the set of vertices and E is the set of edges, does G contain a Hamiltonian Cycle?
HAM-CYCLE = { ⟨G⟩ : G is an undirected graph that contains a Hamiltonian cycle }
A Hamiltonian Cycle is a cycle that visits every vertex in the graph exactly once and returns to the starting vertex. This is different from an Eulerian circuit, which traverses every edge exactly once.
🔵

Hamiltonian Cycle

A cycle v₁ → v₂ → ... → vₙ → v₁ where all vertices are distinct and each consecutive pair (including vₙ, v₁) is connected by an edge.

⭕

Hamiltonian Path

Similar, but does not need to return to the start. A Hamiltonian Path visits every vertex exactly once. Also NP-Complete.

📊

Decision vs Optimization

HAM-CYCLE is a decision problem (yes/no). The optimization variant (TSP) asks for the shortest such cycle. Both are NP-hard.

🗺️

Historical Note

Named after Sir William Rowan Hamilton who invented the Icosian Game in 1857 — a puzzle of finding such a cycle on a dodecahedron graph.

2. Proof That HAM-CYCLE ∈ NP
To show a problem is in NP, we must exhibit a polynomial-time verifier — an algorithm that, given a certificate, can verify in polynomial time whether it constitutes a valid solution.
Certificate
The certificate for an instance ⟨G⟩ is a sequence of vertices: ⟨v₁, v₂, ..., vₙ⟩ claimed to be a Hamiltonian cycle.
Polynomial-Time Verification Algorithm:
1
Length Check: Verify that the sequence contains exactly |V| vertices. This takes O(|V|) time.
2
Uniqueness Check: Verify that no vertex appears more than once (all n vertices are distinct). Using a hash set: O(|V|).
3
Coverage Check: Verify that every vertex in G appears in the sequence. O(|V|) with a hash set.
4
Edge Check: For each consecutive pair (vᵢ, vᵢ₊₁) and the wrap-around (vₙ, v₁), verify that the edge exists in E. O(|V|) with adjacency set.
Conclusion
Total verification time: O(|V| + |E|) = polynomial in the size of the input. Therefore HAM-CYCLE ∈ NP. ✓
3. NP-Completeness via Reduction from Vertex Cover
To prove HAM-CYCLE is NP-Complete, we reduce from VERTEX-COVER (a known NP-Complete problem) to HAM-CYCLE in polynomial time.
This means: VERTEX-COVER ≤ₚ HAM-CYCLE
Vertex Cover (Reminder)
Given graph G = (V, E) and integer k, does G have a vertex cover of size k? (A vertex cover is a set S ⊆ V such that for every edge (u,v) ∈ E, at least one of u, v ∈ S.)
Construction (Polynomial Reduction):
Given instance ⟨G, k⟩ of VERTEX-COVER, construct graph G' as follows:
1
Selector Nodes: Add k selector nodes s₁, s₂, ..., sₖ arranged in a path, representing the k vertices we will choose for the cover.
2
Cover Gadget per Edge: For each edge (u,v) in G, create a cover gadget consisting of 12 nodes arranged in 2×6 structure. This gadget ensures the Hamiltonian cycle must "pick a side" — corresponding to either u or v covering this edge.
3
Connection: Each vertex v in G is connected to all gadgets for edges incident to v. Selector nodes connect to the endpoints of all gadgets, allowing the Hamiltonian cycle to traverse them in order.
Forward Direction (VC → Ham Cycle):
Suppose G has a vertex cover S = {u₁, ..., uₖ} of size k. Construct a Hamiltonian cycle in G' as follows: traverse selector nodes s₁ → ... → sₖ in order. For each selector sᵢ assigned to vertex uᵢ, traverse all gadgets for edges incident to uᵢ from left to right, then return. The cover property ensures every gadget is traversed. Thus G' has a Hamiltonian cycle. ✓
Backward Direction (Ham Cycle → VC):
Suppose G' has a Hamiltonian cycle H. Since selector nodes form a path, each must be entered/exited through gadgets. Each gadget traversal commits to one "side" (u or v). Collect the vertex corresponding to each selector's traversal choices — this gives a set of k vertices. The gadget structure ensures every edge of G is "covered" by at least one selected vertex. Hence, this is a valid vertex cover of size k in G. ✓
Complexity of Reduction
• Number of selector nodes: k (polynomial in input)
• Number of gadgets: |E| (one per edge)
• Each gadget has constant size (12 nodes, 14 edges)
• Total: O(k + |E|) nodes and edges → polynomial time ✓

Since VERTEX-COVER is NP-Complete and VERTEX-COVER ≤ₚ HAM-CYCLE, and HAM-CYCLE ∈ NP, we conclude HAM-CYCLE is NP-Complete. □
4. Simplified Explanation (For Presentations)
Imagine you're a travelling salesperson visiting cities. You want to visit every city exactly once and return home. A Hamiltonian Cycle is exactly this tour — a round trip that touches every stop exactly once.
The question is: does such a tour exist for a given map of cities?
Why is it hard? With n cities, you'd need to check n! (factorial) possible orderings. For just 20 cities, that's 2,432,902,008,176,640,000 orderings — impossible even for modern computers!
NP means: if someone hands you an answer, you can check it quickly (in polynomial time). For Hamiltonian Cycle, if someone says "here's the tour: A→B→C→D→A", you just verify each step is a valid road — easy!
NP-Hard means: it's at least as hard as the hardest problems in NP. We prove this by showing that if you could solve HAM-CYCLE efficiently, you could also solve Vertex Cover efficiently — and Vertex Cover is already known to be NP-Hard.
So HAM-CYCLE is NP-Complete: it's in NP (verifiable quickly) and NP-Hard (as hard as the hardest NP problems).
Graph: 4 nodes (0,1,2,3), Edges: 0-1, 1-2, 2-3, 3-0, 0-2
Question: Is there a Hamiltonian Cycle?
Check path: 0 → 1 → 2 → 3 → 0 Edges needed: (0,1)✓ (1,2)✓ (2,3)✓ (3,0)✓ All edges exist → YES, Hamiltonian Cycle found!
Brute force: With 4 nodes, we check (4-1)! = 6 permutations (fixing start node 0). Only 1 check finds the cycle — but in general, most won't.
5. Algorithm Analysis
Brute Force — O(n!)
Generate all permutations of vertices starting from node 0. For each permutation, check if consecutive vertices are connected and if the last vertex connects back to the start.
Permutations to check: (n-1)! For n=10: 362,880 permutations For n=15: 87,178,291,200 permutations For n=20: 1.2 × 10¹⁷ permutations
Held-Karp Dynamic Programming — O(2ⁿ · n²)
Uses bitmask DP. State: dp[S][v] = "can we visit all vertices in set S, ending at vertex v, starting from vertex 0?"
dp[{0}][0] = true (base case) dp[S][v] = OR over all u in S\{v} where (u,v)∈E: dp[S\{v}][u] Answer: dp[V][0] ∧ (n-1, 0) ∈ E (where V = all vertices)
States: 2ⁿ × n   Transitions per state: O(n)   Total: O(2ⁿ · n²)
For n=20: ≈ 400M operations (vs 10¹⁷ for brute force) — dramatically better!

Why This Matters
This reduction shows that if we could solve HAM-CYCLE in polynomial time, we could also solve VERTEX-COVER in polynomial time. Since VERTEX-COVER is NP-Hard, HAM-CYCLE must be NP-Hard too. Combined with HAM-CYCLE ∈ NP, this proves HAM-CYCLE is NP-Complete.

⚠️ Performance Warning
Brute Force is only feasible for n ≤ 10 in practice (10! = 3.6M, but 12! = 479M). For n = 20, brute force is computationally infeasible even on modern supercomputers.

Held-Karp DP is dramatically better — for n = 20, it requires roughly 4×10⁸ operations (vs 10¹⁸ for brute force), but still exponential. In practice, use heuristic/approximation algorithms for large instances.
